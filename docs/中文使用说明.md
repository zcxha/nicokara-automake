# Nicokara 项目中文使用说明

## 1. 这是什么

这个项目会把一段歌曲视频和一份正确歌词，自动加工成带有卡拉 OK 效果的成品视频。

你给它：

- 一个歌曲视频 `mp4`
- 一份歌词文本 `txt`

它会尽量帮你做出：

- 按时间对齐的歌词
- 逐词高亮的卡拉 OK 字幕
- 汉字上方的假名标注
- 最终烧录好的 `mp4`

如果你完全不懂这些术语，也没关系。下面会一边讲怎么用，一边解释这些东西本质上是在做什么。

## 2. 先用一句人话理解整条流程

你可以把这个项目理解成一个“歌词加工流水线”：

1. 先把视频里的声音拆出来
2. 尽量把“人唱的部分”和“伴奏部分”分开
3. 让语音识别模型去听这段人声，大概判断每个词唱在什么时间
4. 再把这些时间信息贴回你提供的正确歌词上
5. 把结果排成卡拉 OK 字幕
6. 最后把字幕重新压回视频里

这里最关键的思想不是“完全相信 AI 识别出来的歌词文本”，而是：

- 用 AI 提供时间信息
- 用你给的官方歌词提供正确文本
- 用对齐算法把两者拼起来

所以这个项目的目标，不是做“从零猜歌词”的工具，而是做“已知正确歌词时，自动做时间对齐和卡拉 OK 成片”的工具。

## 3. 外行先需要知道的术语

### 3.1 `ffmpeg` 是什么

`ffmpeg` 可以理解成一个处理音视频的“万能加工机”。

它能做很多事，比如：

- 从视频里抽音频
- 转码
- 压制字幕
- 改封装格式

本项目里主要用它做两件事：

- 从原视频里提取音频
- 把生成好的字幕烧录回视频

### 3.2 `ffprobe` 是什么

`ffprobe` 是 `ffmpeg` 的配套工具，主要负责“读取媒体文件信息”。

比如：

- 视频宽高
- 时长
- 音轨信息

本项目用它来读视频分辨率，这样字幕排版时才知道画面有多大。

### 3.3 Demucs 是什么

Demucs 是一个音频分离模型。你可以把它理解成“尽量把歌声从混合音频里抠出来”的工具。

它并不是完美地把人声和伴奏分开，但通常能把歌声提得更清楚一些。这样后面的识别模型更容易听清楚歌词。

### 3.4 `whisper-timestamped` 是什么

它是一个基于 Whisper 的语音识别工具，不过这里最重要的不是“识别出什么字”，而是“它能给出词级时间戳”。

也就是它会告诉我们：

- 某个词大约从哪一秒开始
- 大约在哪一秒结束

这对做逐词高亮字幕非常重要。

### 3.5 ASR 是什么

ASR 是 Automatic Speech Recognition，也就是“自动语音识别”。

在这个项目里，可以简单理解成：

- 听音频
- 猜说了什么
- 并标出时间

### 3.6 对齐是什么

“对齐”指的是把两份信息匹配起来。

这里要匹配的是：

- AI 从声音里识别出的词和时间
- 你提供的正确歌词文本

因为 AI 识别出来的文字不一定完全对，但时间往往有参考价值，所以项目会把“时间”借过来，再贴回“正确歌词”。

### 3.7 `SRT` 和 `ASS` 有什么区别

`SRT` 是最常见、最简单的字幕格式，适合普通字幕。

`ASS` 是更强的字幕格式，能表达：

- 更丰富的样式
- 更精细的位置控制
- 卡拉 OK 逐词高亮
- 多层字幕效果

本项目里：

- `SRT` 主要用来看普通行级字幕
- `ASS` 才是卡拉 OK 成片的核心字幕格式

### 3.8 假名、furigana、ruby 是什么

这几个词在这里几乎是同一件事：就是写在汉字上方的小字读音提示。

例如日文歌词里：

- 正文是汉字
- 上面的小假名告诉你怎么读

在字幕领域里，很多系统也会把这种东西叫 `ruby`。

## 4. 这个项目适合什么场景

当你有：

- 一段歌曲视频 `mp4`
- 一份已经校对好的歌词 `txt`

并且你希望得到：

- 自动时间对齐
- 卡拉 OK 高亮字幕
- 假名标注
- 最终压好字幕的视频

就适合使用这个项目。

## 5. 环境准备

### 5.1 先安装项目依赖

根据仓库约定，进入已经激活的虚拟环境后，先执行：

```bash
uv sync
```

这一步是安装项目自己的 Python 依赖。

### 5.2 再安装命令行工具

```bash
uv tool install --with packaging --with setuptools whisper-timestamped
uv tool install --with torchcodec demucs
```

### 5.3 这些安装命令分别在装什么

- `uv sync`
  安装项目声明的 Python 依赖，比如 `fugashi`、`Pillow`、`pykakasi`
- `whisper-timestamped`
  负责语音识别和词级时间戳
- `demucs`
  负责人声分离

### 5.4 如果你不用 GPU

如果你的机器没有可用 CUDA，或者你就是想先稳定跑通，推荐直接装 CPU 版工具环境：

```bash
uv sync
uv tool install --force --torch-backend cpu --with packaging --with setuptools whisper-timestamped
uv tool install --force --torch-backend cpu --with torchcodec demucs
```

这样通常更省心，因为不会再去依赖 CUDA 相关动态库。

### 5.5 如果你想启用 Sudachi

```bash
uv sync --extra sudachi
```

Sudachi 是另一个日语分词和读音分析后端，适合用来替换或补充默认后端。

### 5.6 系统依赖

请确保系统里有：

- `ffmpeg`
- `ffprobe`

如果没有它们，项目就没法读写视频和烧录字幕。

### 5.7 为什么不建议直接对系统 Python 用 `pip3`

因为很多系统 Python 都是操作系统自己管理的。你直接往里面装包，容易把系统环境弄乱，也容易遇到 `externally-managed-environment` 之类的问题。

这个项目更推荐两件事：

- 用 `uv sync` 安装项目依赖
- 用 `uv tool install ...` 安装 CLI 工具

这样环境更独立，也更容易复现。

## 6. 输入文件要求

### 6.1 视频

- 格式建议是 `mp4`
- 视频里要有歌曲原始音轨
- 不要求必须是纯净伴奏，但音质越清楚越好

### 6.2 歌词

- 纯文本 `txt`
- 一行对应一句歌词
- 尽量使用你最终希望显示在画面上的文字
- 建议去掉明显多余的空行

### 6.3 为什么歌词最好“一行一句”

因为后面生成的很多数据都是按“行”组织的。

如果你把多句歌词塞进同一行，系统虽然还能跑，但会让：

- 时间分配更难看
- 行级字幕更难读
- 后续人工检查更费劲

## 7. 最快上手命令

### 7.1 一条命令直接生成成片

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt"
```

或者：

```bash
nicokara-build "input.mp4" "lyrics.txt"
```

默认会生成一个类似 `input.nicokara/` 的目录。

### 7.2 第一次更推荐先不烧录

如果你是第一次处理一首歌，更推荐先跑：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" --skip-burn
```

这样它会先把中间结果和字幕文件生成出来，但不会重新压视频。你可以先检查结果是否满意，再决定是否最终烧录。

这是因为整条流程里最耗时的通常是：

- 人声分离
- 语音识别
- 最终视频烧录

先跳过最后一步，调试会快很多。

## 8. 一次完整运行时，程序到底在做什么

以这条命令为例：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt"
```

背后大致会发生下面这些事：

1. 程序先创建输出目录，比如 `input.nicokara/`
2. 用 `ffmpeg` 从原视频里提取音频，生成 `input.audio.wav`
3. 用 Demucs 尽量把歌声从混合音频里分出来，生成 `input.vocals.wav`
4. 用 `whisper-timestamped` 对人声音频做识别，生成 `*.words.json`
5. 把识别结果和你提供的正确歌词做对齐，生成 `*.nicokara.json`
6. 根据对齐结果生成普通字幕 `*.srt`
7. 根据对齐结果生成卡拉 OK 字幕 `*.ass`
8. 最后用 `ffmpeg` 把 `ASS` 字幕烧录回原视频

这里有一个很重要的理解方式：

- `*.words.json` 更像“机器听出来的大概时间信息”
- `*.nicokara.json` 更像“修正后的正式中间结果”
- `*.karaoke.ass` 更像“最终用于画面渲染的字幕脚本”

## 9. 输出目录里会有什么

假设输入视频叫 `input.mp4`，常见输出如下：

- `input.audio.wav`
  从视频提取出的音频
- `input.vocals.wav`
  人声分离结果
- `input.vocals.wav.words.json`
  识别模型输出的词级时间戳
- `input.nicokara.json`
  正确歌词和时间信息融合后的核心结果
- `input.nicokara.srt`
  普通行级字幕
- `input.karaoke.ass`
  带高亮和假名的卡拉 OK 字幕
- `input.nicokara.mp4`
  最终成片

## 10. 最常用参数

### 10.1 指定输出目录

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --output-dir "build/my_song"
```

### 10.2 指定最终视频输出路径

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --output-video "release/my_song.nicokara.mp4"
```

### 10.3 指定 Whisper 模型

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --whisper-model medium
```

可以把它理解成“识别模型规格”。一般来说：

- 模型越大，通常越慢、资源占用越高
- 模型越小，通常越快，但在复杂场景下可能更容易出错

如果机器性能一般，先用默认的 `small` 就够了。

### 10.4 指定识别语言

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --language ja
```

这里的作用是给识别模型一个“语言提示”，让它更倾向按日语去理解声音。

### 10.5 指定设备

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --whisper-device cpu
```

如果你的 CUDA 环境可用，也可以：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --whisper-device cuda
```

### 10.6 开启 Whisper VAD

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --whisper-vad
```

VAD 可以理解成“语音活动检测”，也就是让模型尽量先判断“哪里有人声、哪里没有”。

它在有些噪声场景里能帮助识别，但不是所有歌曲都会更好。

### 10.7 指定 Demucs 模型

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --demucs-model htdemucs_ft
```

这决定了做人声分离时使用哪一个 Demucs 模型。

### 10.8 强制重跑

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --force
```

当已有缓存结果时，项目通常会复用旧文件。加上 `--force` 就是告诉它“不要偷懒，重新生成”。

### 10.9 只生成字幕和 JSON，不烧录视频

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --skip-burn
```

适合你先检查：

- 时间对齐是否自然
- 假名是否正确
- 卡拉 OK 高亮边界是否合理

### 10.10 指定假名后端

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --reading-backend auto
```

可选值：

- `auto`
- `fugashi`
- `sudachi`
- `pykakasi`

默认推荐 `auto`。

你可以把“假名后端”理解成“系统拿来分析日文文字结构和读音的工具箱”。

### 10.11 指定 Sudachi 切分模式

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --reading-backend sudachi \
  --reading-split-mode C
```

这会影响 Sudachi 把句子切成多细的单位。

### 10.12 指定外部 furigana 资源

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --furigana-resource "/path/to/JmdictFurigana.json.gz"
```

这里的资源可以理解成“额外的标准读音词典”。当系统遇到复杂汉字词时，这类资源有助于给出更准确的假名。

### 10.13 指定人工读音覆写

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --reading-overrides "ruby_overrides.json"
```

这相当于你手动告诉系统：

- 某个词就按这个读音来
- 不要再猜

当歌词里出现人名、地名、特殊艺名或故意写法时，这个功能很有用。

## 11. 为什么日文没有空格，也还能逐词高亮

这是很多外行第一次接触时最困惑的点。

中文和日文里，文本经常不像英文那样天然用空格隔开词。所以程序不能只靠空格判断“哪里是一个词”。

这个项目会做两件事：

- 如果歌词里本来就有空格，会优先尊重这些空格
- 如果没有空格，会使用日语分词和读音后端去推测更合理的切分

所以即使一整句没有空格，系统也不一定会把整句只亮一次，而是会尽量拆成更自然的高亮单元。

## 12. 为什么字体会影响假名位置

这也是一个很常见但很反直觉的问题。

现在项目里假名的位置，不是靠“数几个字”算的，而是直接让 `ffmpeg/libass` 先把正文排一遍，再根据实际渲染结果去推导每个字符边界。

所以这里真正依赖的是：

- `ffmpeg/libass` 实际怎么排这行字
- 它实际加载到的是哪一个字体文件

而不是简单地看字符个数。

也就是说：

- 同样 4 个字
- 换一个字体
- 它们占的像素宽度可能就不同

本项目现在会先用 `libass` 测量整行正文的字符边界，再把假名放到对应片段的中心。

所以如果：

- 测量阶段用的是字体 A
- 最终烧录阶段实际回退成了字体 B

就很可能出现假名错位。

不过和旧版本不同的是：

- 旧版本更像是“Python 先估算，再交给 `ffmpeg` 去画”
- 现在更像是“直接按 `ffmpeg/libass` 的实际排版结果来算”

所以只要字体一致，新版通常会比旧版更贴近最终烧录效果。

### 12.1 推荐的字体使用方式

最简单的方案是：

1. 把字体文件放到仓库根目录 `fonts/`
2. 确认字体族名和项目里使用的名字一致
3. 再运行项目

仓库里已经准备了示例字体：

- 文件：`fonts/NotoSansCJKjp-Regular.otf`
- 字体族名：`Noto Sans CJK JP`

### 12.2 相关环境变量

```bash
export NICOKARA_FONT_DIR="/path/to/fonts"
export NICOKARA_KARAOKE_FONT_NAME="Noto Sans CJK JP"
export NICOKARA_RUBY_FONT_NAME="Noto Sans CJK JP"
```

可以这样理解：

- `NICOKARA_FONT_DIR`
  告诉项目去哪里找字体文件
- `NICOKARA_KARAOKE_FONT_NAME`
  告诉项目正文应该用哪个字体族名
- `NICOKARA_RUBY_FONT_NAME`
  告诉项目假名应该用哪个字体族名

## 13. 复用已有中间结果

### 13.1 已经有人声音频

如果你已经有 `vocals.wav`，可以跳过 Demucs：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --vocals "song.vocals.wav"
```

### 13.2 已经有 Whisper JSON

如果你已经有 `*.words.json`，可以直接跳过分离和识别：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --asr-json "song.vocals.wav.words.json"
```

这通常是反复微调歌词和字幕时最快的方式。

## 14. 只想反复调对齐和字幕

当前没有单独的“只对齐”命令，但最方便的办法是复用已有 ASR JSON 并加上 `--skip-burn`：

```bash
python3 build_nicokara_video.py "input.mp4" "lyrics.txt" \
  --asr-json "song.vocals.wav.words.json" \
  --skip-burn
```

这样会重新生成：

- `*.nicokara.json`
- `*.nicokara.srt`
- `*.karaoke.ass`

但不会重新压制成最终视频。

## 15. 歌词书写建议

为了得到更自然的结果，建议：

- 一行就是一句
- 文本尽量和最终希望显示在画面上的歌词一致
- 不要把多段歌词挤进同一行
- 如果你想手动指定高亮边界，可以在适当位置加空格

如果不加空格也没关系，项目会尽量自动切分。

## 16. Ruby 诊断工具

生成 `*.nicokara.json` 后，可以用下面的命令筛出值得人工复核的词：

```bash
nicokara-ruby-diagnostics song.nicokara.json > suspicious_ruby.json
```

这个工具默认会重点标出：

- 缺少 `ruby_text` 的词
- `coverage` 太低的词
- `ruby_source == "backend"` 的词

你也可以顺手生成一个人工覆写模板：

```bash
nicokara-ruby-diagnostics \
  song.nicokara.json \
  --overrides-template ruby_overrides.json
```

这一步的意义可以理解成：先让程序自己做，再把“可能不稳的地方”集中拎出来给人复查。

## 17. 常见问题

### 17.1 报错找不到 `whisper_timestamped`

执行：

```bash
uv tool install --with packaging --with setuptools whisper-timestamped
```

### 17.2 报错找不到 `demucs`

执行：

```bash
uv tool install --force --with torchcodec demucs
```

### 17.3 报错找不到 `ffmpeg` 或 `ffprobe`

这说明系统层面的音视频工具还没安装，需要先安装 FFmpeg。

### 17.4 输出字幕对齐不够理想

先检查：

- 歌词文本是否正确
- 歌词是否一行一句
- 音频里人声是否过于模糊
- 是否需要更大的 Whisper 模型
- 是否需要补 `--reading-overrides`

推荐排查顺序：

1. 先用 `--skip-burn` 生成中间文件
2. 检查 `*.nicokara.json`
3. 检查 `*.karaoke.ass`
4. 运行 `nicokara-ruby-diagnostics *.nicokara.json`
5. 再决定是改歌词、改模型、改覆写还是改字体

### 17.5 首次运行很慢

这是正常的，因为首次运行时可能会发生：

- Demucs 下载模型
- Whisper 下载模型
- 默认假名链路下载 `JmdictFurigana / JmnedictFurigana`
- 最终视频烧录本身耗时较长

第二次复用缓存后通常会快很多。

如果你需要强制离线运行，可以先准备好本地 furigana 资源，或者设置：

```bash
export NICOKARA_DISABLE_AUTO_FURIGANA_DOWNLOAD=1
```

## 18. 推荐使用流程

如果你是第一次处理一首歌，推荐这样做：

1. 准备好 `mp4` 和干净的歌词 `txt`
2. 执行 `python3 build_nicokara_video.py "input.mp4" "lyrics.txt" --skip-burn`
3. 检查 `*.nicokara.json` 和 `*.karaoke.ass`
4. 执行 `nicokara-ruby-diagnostics *.nicokara.json --overrides-template ruby_overrides.json`
5. 如果读音、边界或文本不满意，就修改歌词或覆写文件后重跑
6. 最后去掉 `--skip-burn` 生成成片
