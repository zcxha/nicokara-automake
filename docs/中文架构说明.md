# Nicokara 项目中文架构说明

## 1. 文档目标

这份文档不是单纯列文件名，而是希望解释三件事：

- 这个项目整体在解决什么问题
- 每个阶段在计算机里到底做了什么
- 为什么系统要这样拆模块，而不是把所有逻辑揉成一个脚本

如果你是第一次接触音视频、ASR、字幕格式或日语读音处理，也可以顺着读下去。这里会尽量用“工程上真实但不故作高深”的方式解释原理。

## 2. 项目目标

这个项目的目标是：

把一段带画面的歌曲视频和一份正确歌词，自动加工成带有卡拉 OK 式逐词高亮和假名标注的ニコカラ成品视频。

输入：

- 一个 `mp4` 视频
- 一份官方或人工校对后的歌词 `txt`

输出：

- 词级对齐结果 `*.nicokara.json`
- 行级字幕 `*.nicokara.srt`
- 卡拉 OK 字幕 `*.karaoke.ass`
- 最终成品 `*.nicokara.mp4`

## 3. 核心设计思想

### 3.1 这个系统真正依赖的，不是“完美识别”

很多人第一次看到这类工具，会直觉认为它的工作方式是：

- 听音频
- 识别出歌词
- 直接拿识别文本做字幕

但这个项目并不是这么设计的。

它真正采用的是一种“时间和文本分离”的思路：

- 语音识别主要负责提供时间线索
- 正确歌词负责提供最终文本
- 对齐算法负责把时间信息贴回正确文本

这样做的原因很现实：

- 语音识别很擅长给出大概时间
- 但歌词文本本身常常会识别错
- 尤其是同音字、艺名、古风写法、拖长音、混响和和声场景

所以系统不直接相信“机器认出来的字”，而是更重视“机器给出的时间锚点”。

### 3.2 这个项目本质上是一个流水线系统

它不是一个单点算法，而是一串依次发生的处理阶段。

这些阶段分别擅长不同问题：

- 媒体工具擅长处理音视频文件
- 分离模型擅长突出人声
- 识别模型擅长给出时间戳
- 对齐算法擅长融合时间和正确文本
- 字幕渲染层擅长把这些数据变成可显示的画面

把它们拆开，最大的好处是：

- 每一步都可以单独调试
- 中间产物可以复用
- 识别、对齐、渲染可以分别优化

## 4. 总体数据流

整体流水线如下：

```text
MP4
  -> ffmpeg 提取音频
  -> Demucs 分离人声
  -> whisper_timestamped 词级识别
  -> 正确歌词与识别结果对齐
  -> 生成 SRT / ASS 字幕
  -> ffmpeg 烧录回视频
  -> ニコカラ成品 MP4
```

如果用户已经有中间产物，也可以从中途接入：

- 已有 `vocals.wav` 时，跳过分离
- 已有 `*.words.json` 时，跳过分离和识别

这说明系统并不把“从头跑完”写死，而是把中间阶段也当成正式接口。

## 5. 为什么每一步都存在

### 5.1 为什么先提取音频

因为后面的人声分离和语音识别都主要处理音频，而不是直接处理视频画面。

从工程角度说，先把音频变成一个稳定的 `wav` 文件，后面很多工具更容易复用，也更容易排查问题。

### 5.2 为什么要做人声分离

歌曲音频里通常混在一起的东西很多：

- 主唱
- 和声
- 乐器
- 混响
- 环境声

语音识别模型理论上可以直接听整首歌，但伴奏越强、混响越重，识别往往越不稳定。

人声分离并不是为了得到“完美无损的人声轨”，而是为了给识别模型一个更容易听清楚歌词的输入。

### 5.3 为什么还要语音识别

因为歌词文本本身不带时间。

你给项目一份 `txt`，它只能知道“歌词长什么样”，但不知道“第几个字在第几秒唱出来”。

语音识别模型的价值就在这里：

- 它能从声音里估计出时间
- 它给系统提供逐词或近似逐词的时间锚点

### 5.4 为什么还需要对齐

因为语音识别的输出并不完全可信。

识别文本可能会出现：

- 错字
- 漏字
- 把歌词写法听成另一个同音词
- 把拖音拆错

但官方歌词通常才是你真正想展示给观众看的文本。

所以系统必须做一次“融合”：

- 保留识别结果里的时间价值
- 保留官方歌词里的文本正确性

这一步就是对齐层的意义。

### 5.5 为什么最终生成 `ASS`

因为普通字幕格式只能表达“哪一行在什么时间出现”，很难表达真正的卡拉 OK 效果。

而 `ASS` 支持：

- 逐词高亮
- 更细的定位
- 多层文本事件
- 假名和正文分层显示

本项目当前默认的 `ASS` 布局也会额外做一层“ニコカラ风格”的排版约束：

- 主歌词和假名会比普通字幕更大
- 字幕区整体上移，避免死贴最底边
- 相邻歌词句子交替放在左上和右下两个显示槽位
- 下一句可以在上一句开始时提前显示，形成更像成品ニコカラ的双句候场效果

所以从“能播放字幕”到“能做卡拉 OK 成片”，中间差的其实就是一个更强的渲染描述层。

## 6. 模块划分

### 6.1 CLI 入口

文件：[build_nicokara_video.py](/home/zcx/source/nicokara/build_nicokara_video.py)

职责：

- 解析命令行参数
- 组织用户输入
- 调用主流水线
- 把常见异常转换成用户友好的报错

这一层的作用不是做核心算法，而是把“用户说的话”翻译成“程序内部参数”。

可以把它理解成前台接待。

### 6.2 主流水线编排

文件：[pipeline.py](/home/zcx/source/nicokara/nicokara/pipeline.py)

职责：

- 决定执行顺序
- 管理输出目录和中间文件路径
- 决定什么时候复用缓存
- 串联媒体处理、识别、对齐、字幕生成和烧录

这一层是整个系统的调度中心。

如果把项目比作工厂，`pipeline.py` 更像“生产线总控”，而不是某一台具体机器。

### 6.3 媒体处理层

文件：[media.py](/home/zcx/source/nicokara/nicokara/media.py)

职责：

- 调用 `ffmpeg` 提取音频
- 调用 `ffprobe` 读取视频分辨率
- 调用 `ffmpeg` 把 `ASS` 烧录回视频
- 统一封装外部命令执行和错误处理

这一层只关心“怎么和系统里的媒体工具打交道”，不关心歌词内容。

### 6.4 对齐层

文件：[alignment.py](/home/zcx/source/nicokara/nicokara/alignment.py)

职责：

- 读取 `whisper-timestamped` 的词级 JSON
- 读取用户提供的正确歌词
- 把文本变成便于比较的读音序列
- 做字符级单调对齐
- 再把结果聚合成行级和词级时间

这是项目里最核心的算法层之一。

“单调对齐”可以很直观地理解成：

- 识别时间一定是沿着歌曲从前往后走
- 歌词顺序也一定是从前往后走
- 所以匹配时不允许时间倒流，也不允许把后面的字匹配到前面的时间上

这类约束让对齐结果更符合真实演唱过程。

### 6.5 字幕生成层

文件：

- [ass.py](/home/zcx/source/nicokara/nicokara/ass.py)
- [srt.py](/home/zcx/source/nicokara/nicokara/srt.py)
- [fonts.py](/home/zcx/source/nicokara/nicokara/fonts.py)

职责：

- `srt.py` 生成普通行级字幕
- `ass.py` 生成卡拉 OK 高亮和假名排版
- `fonts.py` 负责字体发现、字体匹配和烧录阶段字体环境准备

这一层做的不是“算法推理”，而是“把已有结果变成屏幕上看得见的字幕”。

### 6.6 文本与读音处理层

文件：[convert.py](/home/zcx/source/nicokara/nicokara/convert.py)

职责：

- 提供统一的日语读音处理接口
- 支持 `fugashi`、`SudachiPy`、`pykakasi` 等后端
- 为对齐阶段提供标准化的读音表示
- 为没有空格的日文歌词提供自动切分
- 生成 `ruby_parts` 供字幕层定位假名

这一层相当于“语言学接口适配层”。

它把不同读音后端的差异，尽量收敛成系统内部统一的数据形式。

### 6.7 诊断层

文件：[ruby_diagnostics.py](/home/zcx/source/nicokara/nicokara/ruby_diagnostics.py)

职责：

- 扫描 `*.nicokara.json`
- 找出值得人工复查的假名项
- 输出可筛查的诊断结果
- 生成覆写模板

这层很重要，因为工程上真正可靠的系统，不是“假装永远正确”，而是“能把不确定的地方单独标出来给人看”。

## 7. 一次完整运行会发生什么

以：

```bash
python3 build_nicokara_video.py input.mp4 lyrics.txt
```

为例，内部大致会经历：

1. CLI 读取参数
2. `pipeline.py` 创建输出目录和目标路径
3. `media.py` 从视频提取 `input.audio.wav`
4. `pipeline.py` 调用 Demucs 生成人声轨
5. `pipeline.py` 调用 `whisper-timestamped` 得到 `*.words.json`
6. `alignment.py` 把 ASR 结果和正确歌词融合成 `*.nicokara.json`
7. `srt.py` 生成 `*.nicokara.srt`
8. `ass.py` 生成 `*.karaoke.ass`
9. `media.py` 调用 `ffmpeg` 烧录字幕，得到最终 `*.nicokara.mp4`

其中 Demucs 这一步在当前实现里会优先经过项目自己的兼容运行器。
这样做的原因是：新版本 `torchaudio` 在保存音频时可能转而依赖 `torchcodec`，而 Windows 环境里这一步偶尔会因为动态库兼容问题失败。
兼容运行器不会改动 Demucs 的分离模型本身，只是把最终 `vocals.wav` 的写出方式替换成更稳的本地实现。

这串步骤之间不是简单文件拷贝，而是一次次“信息压缩和重表达”：

- 原始视频里信息很多，但对歌词处理真正有用的是音频
- 原始音频里信息很多，但对识别真正有用的是更清晰的人声
- 识别结果里信息很多，但对最终字幕真正有用的是时间锚点
- 最终字幕里既要有文本，也要有排版坐标和动画信息

## 8. 核心数据结构

### 8.1 `PipelineArtifacts`

定义位置：[pipeline.py](/home/zcx/source/nicokara/nicokara/pipeline.py)

它描述一次流水线运行后最主要的产物路径，比如：

- 提取音频
- 人声音频
- ASR JSON
- 对齐 JSON
- SRT
- ASS
- 最终 MP4

它的意义在于：让“整条流水线最终产出了什么”有一个稳定的结构化表达。

### 8.2 `*.nicokara.json`

这是整条流程最关键的共享中间格式之一。

主要包含：

- `source`
  输入来源信息
- `alignment`
  对齐统计信息，以及实际生效的 `reading_backend`
- `lines`
  每一行歌词的文本、时间和覆盖率
- `words`
  每个高亮单元的文本、时间、覆盖率、`ruby_text`、`ruby_parts` 和 `ruby_source`

你可以把它理解成：

- 一边站着“正确歌词”
- 一边站着“从音频里估出来的时间”
- 中间由对齐层把二者融合成一个可复用的结构

后续：

- `SRT` 生成器读它
- `ASS` 生成器读它
- 诊断工具也读它

所以它是一个典型的“流水线内部标准中间件格式”。

## 9. 关键原理解释

### 9.1 为什么无空格日文也能做逐词高亮

因为系统并不只依赖空格。

它会结合：

- 形态学分词后端
- 读音信息
- 词典资源
- 动态规划式的切分和对齐逻辑

来推断更合理的高亮单位。

换句话说，系统真正做的是：

- 尽量判断一句歌词哪些字符应该属于同一个“演唱单元”
- 再为这些单元分配时间

### 9.2 为什么假名能贴到对应汉字上

因为系统不是把整行假名简单地居中丢上去，而是做了更细的事情：

- 先得到每个词的 `ruby_parts`
- 再用 `ffmpeg/libass` 直接测量整行正文不同字符边界的位置
- 计算每个假名片段对应的水平中心
- 为每个假名片段生成独立定位事件

这本质上是一种“文本宽度驱动的几何排版”。

它不是在理解文字语义，而是在计算“这些字被 `libass` 画到屏幕上之后，边界和中心点到底在哪”。

### 9.3 为什么字体不一致会导致错位

因为宽度测量依赖字体，而字幕烧录也依赖字体。

如果两边不是同一字体，系统看到的“文本宽度”就会变。

于是：

- 算出来的位置是基于字体 A
- 实际画出来却用了字体 B

假名自然会漂移。

这也是为什么项目额外引入了 `fonts.py` 来统一字体发现逻辑，并让测量阶段与最终烧录阶段尽量共用同一套字体文件。

### 9.4 为什么需要诊断工具

因为假名读音并不是一个永远可以自动做对的问题。

现实里会遇到：

- 生僻读法
- 人名地名
- 歌词故意使用非常规写法
- 词典里没有的写法

所以好的工程做法不是假设“自动系统一定全对”，而是：

- 自动先做一遍
- 把风险点集中标出来
- 让人有机会最小成本地复查

## 10. 外部依赖在架构中的角色

项目依赖三类外部能力：

- `ffmpeg / ffprobe`
  负责音视频文件处理和烧录
- `demucs`
  负责人声分离
- `whisper-timestamped`
  负责词级时间戳识别

Python 侧核心依赖：

- `fugashi[unidic-lite]`
- `Pillow`
- `pykakasi`

可选依赖：

- `SudachiPy`
- `sudachidict_core`

运行时可能自动下载的资源：

- `JmdictFurigana`
- `JmnedictFurigana`

从系统设计上说，这些依赖各自提供一种专业能力，而本项目本身负责把这些能力组织成一条稳定、可复用的生产线。

## 11. 当前工程边界

当前实现已经支持：

- `mp4` 输入
- 正确歌词纠正
- 逐词高亮 `ASS`
- 片段级假名定位
- 最终烧录 `mp4`

当前仍然保留的边界包括：

- 字幕样式目前是工程内置样式，不是完整主题系统
- 极端歌词写法仍可能需要 `--reading-overrides`
- 当识别局部严重失败时，部分时间仍会依赖插值
- 首次默认假名链路可能需要联网下载资源
