# Nicokara 项目中文架构说明

## 1. 项目目标

这个项目的目标是把一段带画面的歌曲视频和一份正确歌词，自动加工成带有 KTV 式逐词高亮字幕的ニコカラ成品视频。

输入：

- 一个 `mp4` 视频
- 一份官方或人工校对后的歌词 `txt`

输出：

- 词级对齐结果 `*.nicokara.json`
- 行级字幕 `*.nicokara.srt`
- KTV 字幕 `*.karaoke.ass`
- 成品视频 `*.nicokara.mp4`

## 2. 总体数据流

整体流水线如下：

```text
MP4
  -> ffmpeg 提取音频
  -> Demucs 分离人声
  -> whisper_timestamped 词级识别
  -> 歌词纠正与时间对齐
  -> ASS KTV 字幕生成
  -> ffmpeg 烧录回原视频
  -> ニコカラ成品 MP4
```

如果用户已经有中间产物，也可以从中途接入：

- 已有 `vocals.wav` 时，跳过分离
- 已有 `*.words.json` 时，跳过分离和识别

## 3. 模块划分

### 3.1 CLI 入口

文件：[build_nicokara_video.py](../build_nicokara_video.py)

职责：

- 解析命令行参数
- 调用主流水线
- 打印最终产物路径
- 把底层异常转成用户友好的报错

### 3.2 主流水线编排

文件：[pipeline.py](../nicokara/pipeline.py)

职责：

- 决定整条流水线的执行顺序
- 管理中间文件路径和输出目录
- 决定何时复用缓存、何时重新生成
- 连接音频处理、ASR、歌词对齐、字幕生成和烧录

这里是整个系统的“调度层”。

### 3.3 媒体处理

文件：[media.py](../nicokara/media.py)

职责：

- 调用 `ffmpeg` 提取音频
- 调用 `ffprobe` 读取视频分辨率
- 调用 `ffmpeg` 把 ASS 烧录回视频
- 提供统一的外部命令执行和错误包装

这一层只关心“怎么和系统工具打交道”，不关心歌词逻辑。

### 3.4 歌词对齐

文件：[alignment.py](../nicokara/alignment.py)

职责：

- 读取 Whisper 的词级时间戳 JSON
- 读取用户提供的正确歌词
- 把歌词转成可对齐的假名序列
- 做字符级单调对齐
- 重新聚合成行级和词级时间结果

这是项目里最核心的算法层。

### 3.5 字幕生成

文件：

- [ass.py](../nicokara/ass.py)
- [srt.py](../nicokara/srt.py)

职责：

- `srt.py` 负责常规行级字幕
- `ass.py` 负责 KTV 高亮字幕

其中 `ass.py` 会把每一行拆成多个高亮单元，并用 `{\k...}` 标签生成卡拉 OK 动画。

### 3.6 假名转换与歌词切分

文件：[convert.py](../nicokara/convert.py)

职责：

- 使用 `pykakasi` 做假名转换
- 为对齐提供标准化文本
- 为无空格日文歌词提供自动切分能力

## 4. 一次完整运行会发生什么

以 `python3 build_nicokara_video.py input.mp4 lyrics.txt` 为例：

1. CLI 收到 `video` 和 `lyrics`
2. `pipeline.py` 创建输出目录 `input.nicokara/`
3. `media.py` 用 `ffmpeg` 从原视频提取 `input.audio.wav`
4. `pipeline.py` 调用 Demucs，把音频分离成 `input.vocals.wav`
5. `pipeline.py` 调用 `whisper_timestamped`，得到 `input.vocals.wav.words.json`
6. `alignment.py` 把 ASR 结果和正确歌词对齐，生成 `input.nicokara.json`
7. `srt.py` 产出行级 `input.nicokara.srt`
8. `ass.py` 产出 KTV `input.karaoke.ass`
9. `media.py` 用 `ffmpeg + libass` 烧录字幕，得到 `input.nicokara.mp4`

## 5. 核心数据结构

### 5.1 PipelineArtifacts

定义位置：[pipeline.py](../nicokara/pipeline.py)

它描述一次流水线运行后的主要产物路径，例如：

- 提取出的音频
- 人声音频
- ASR JSON
- 对齐 JSON
- SRT
- ASS
- 最终 MP4

### 5.2 nicokara JSON

`*.nicokara.json` 是对齐阶段的核心输出，主要包含：

- `source`
  记录输入来源
- `alignment`
  记录整体匹配统计
- `lines`
  每一行的文本、时间和覆盖率
- `words`
  每个高亮单元的文本、时间和覆盖率

这个 JSON 是“字幕生成层”和“外部调试分析”的共享中间格式。

## 6. 几个重要设计点

### 6.1 为什么先分离人声再识别

因为原视频里的伴奏、环境声和混响会明显影响识别质量。先做人声分离，能提高 Whisper 的歌词识别稳定性。

### 6.2 为什么要用“正确歌词”反校正 ASR

Whisper 能给出较好的时间信息，但对日文歌词里的同音字、罕见词和拖音并不总是可靠。

所以这里采用的策略不是“直接相信识别文本”，而是：

- 用 Whisper 提供时间锚点
- 用用户提供的正确歌词提供最终文本
- 再通过字符级对齐把两者融合

这样能同时兼顾“文本正确”和“时间可用”。

### 6.3 为什么无空格歌词也能高亮

日文歌词经常没有空格。如果完全依赖空格切词，会导致一整句只亮一次。

现在项目的处理方式是：

- 有空格时，优先相信用户的空格边界
- 无空格时，使用 `pykakasi` 的转换结果切成更细的高亮单元

这让 KTV 字幕观感更接近实际卡拉 OK。

### 6.4 为什么输出 ASS 而不是只输出 SRT

因为：

- `SRT` 适合普通字幕
- `ASS` 才适合做逐词高亮和更丰富的排版

最终的 KTV 效果主要依赖 `ASS`。

## 7. 外部依赖

项目依赖三类外部能力：

- `ffmpeg / ffprobe`
  负责媒体读写与烧录
- `Demucs`
  负责人声分离
- `whisper_timestamped`
  负责词级时间戳 ASR

Python 侧还需要：

- `pykakasi`

## 8. 当前边界

当前实现已经支持：

- MP4 输入
- 正确歌词纠正
- 逐词高亮 ASS
- 最终烧录 MP4

当前仍然保留的工程边界：

- 字幕样式目前是工程内置样式，不是完整的主题系统
- 分词策略对日文已经比空格更好，但还不是专业形态学分词器
- 对极端识别失败的片段，时间分配仍然会依赖插值
